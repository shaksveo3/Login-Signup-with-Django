Signup Process - Step By Step

Complete Signup Flow

1. User visits http://localhost:8000/ ‚Üì 2. Django calls signup_view() with GET request ‚Üì 3. signup_view() returns signup.html (empty form) ‚Üì 4. User sees form with fields: - Username - Email - Password - Confirm Password ‚Üì 5. User fills form and clicks "Sign Up" ‚Üì 6. Form submitted with POST method ‚Üì 7. Django calls signup_view() with POST request ‚Üì 8. Backend processes data and validates ‚Üì 9. Views.py - Signup Function

from django.shortcuts import render, redirect from django.contrib.auth.models import User from django.contrib import messages def signup_view(request): # Check if form was submitted if request.method == 'POST': # Get form data username = request.POST['username'] email = request.POST['email'] password = request.POST['password'] password2 = request.POST['password2'] # Step 1: Check passwords match if password == password2: # Step 2: Check username doesn't exist if User.objects.filter(username=username).exists(): messages.error(request, 'Username already exists') return redirect('signup') # Step 3: Check email doesn't exist elif User.objects.filter(email=email).exists(): messages.error(request, 'Email already exists') return redirect('signup') # Step 4: Create user else: user = User.objects.create_user( username=username, email=email, password=password ) user.save() messages.success(request, 'Account created! Please login.') return redirect('login') else: messages.error(request, 'Passwords do not match') return redirect('signup') # Show empty form for GET request return render(request, 'signup.html')

Password Hashing Explained

Plain Password: "hello123" ‚Üì Django's Hashing Algorithm (PBKDF2) ‚Üì Stored in Database: "pbkdf2_sha256$260000$abc123def456$..." ‚Üì KEY POINT: Cannot be reversed! You cannot get "hello123" back from the hash. When user logs in, Django hashes their password and compares the hashes instead!

Authentication Process Explained

User enters: - username = "john" - password = "hello123" ‚Üì authenticate() function: ‚îú‚îÄ Search database for user "john" ‚îú‚îÄ Get his stored password hash ‚îú‚îÄ Hash the entered password "hello123" ‚îî‚îÄ Compare: entered_hash == stored_hash ? ‚îú‚îÄ YES ‚Üí Return user object ‚úì ‚îî‚îÄ NO ‚Üí Return None ‚úó ‚Üì if user is not None: login(request, user) ‚Üê Create session return redirect('welcome') else: Show error message

What Is authenticate()?

authenticate(request, username, password):
‚Ä¢ Searches database for user with given username
‚Ä¢ Checks if password matches (via hashing)
‚Ä¢ Returns User object if match, None if not

What Is login()?

login(request, user):
‚Ä¢ Creates a session for the user
‚Ä¢ Sends session cookie to browser
‚Ä¢ Django remembers this user for future requests

Sessions Explained

SESSION = Server's way of remembering users Without Session: - Request 1: "Who are you?" ‚Üí User says "I'm john" - Request 2: "Who are you?" ‚Üí User says "I'm john" (AGAIN!) - Annoying! With Session: - Request 1: "Who are you?" ‚Üí "I'm john" ‚Üí Server creates session - Request 2: Browser sends session ID in cookie ‚Üí Server checks: "Ah, this is john's session" - Much better! When user logs out: - Session is destroyed - Cookie is deleted - Next request: User is anonymous again

Security Concepts For Beginners

1. Password Hashing

‚úÖ GOOD - Hashing

Password: "hello123" ‚Üì Hash: "pbkdf2_sha256$..." ‚Üì ‚Ä¢ One-way (can't reverse) ‚Ä¢ Unique for each password ‚Ä¢ Same password = same hash ‚Ä¢ Cannot retrieve original

‚ùå BAD - Plain Text

Password: "hello123" ‚Üì Stored: "hello123" ‚Üì ‚Ä¢ Anyone with database access sees it ‚Ä¢ Used in multiple sites = compromised ‚Ä¢ Cannot verify without comparing ‚Ä¢ Major security risk!

2. CSRF Token Protection

CSRF = Cross-Site Request Forgery Hacker's Website Your Website ‚Üì ‚Üì "Click me" ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ[stolen cookie]‚îÄ‚Üí Delete account ‚Üì Vulnerable! With CSRF Token: "Click me" ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ[stolen cookie]‚îÄ‚Üí Needs CSRF token ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ[no token]‚îÄ‚îÄ‚Üí ‚úó Rejected! ‚úì Protected!

3. Session Security

Session Cookie: Contains session ID, not password
Secure: Session ID is random and expires
HttpOnly: JavaScript cannot access (JavaScript can't steal it)

4. SQL Injection Prevention

‚úÖ SAFE - Django ORM

User.objects.filter( username=username ) ‚Ä¢ Uses parameterized queries ‚Ä¢ Django escapes dangerous characters ‚Ä¢ No SQL injection possible

‚ùå DANGEROUS - Raw SQL

SELECT * FROM auth_user WHERE username='{username}' If username = "' OR '1'='1" Result: Gets ALL users! ‚Ä¢ Always vulnerable ‚Ä¢ Never do this!

5. @login_required Protection

@login_required decorator ensures:
‚Ä¢ Only authenticated users can access view
‚Ä¢ Anonymous users redirected to login
‚Ä¢ Prevents unauthorized data access

Common Questions & Answers

Q: What happens if I don't use @login_required?

A: Anyone (logged in or not) can access the page. The welcome page would show "Welcome AnonymousUser!" to strangers.

Q: Why hash passwords instead of encrypting?

A: Encryption can be reversed (bad). Hashing is one-way. Even Django staff can't recover original passwords - only users can!

Q: What if user enters wrong password?

A: authenticate() returns None. View shows error message. User redirected to login to try again.

Q: How long does a session last?

A: Default 2 weeks of inactivity. Configurable in settings.py. Logout immediately deletes session.

Q: Can I use Django's User model for my own fields?

A: Create custom model extending User, or create separate model linked to User (UserProfile). More advanced topic.

Q: How do I handle "Forgot Password"?

A: Generate secure token, send email with reset link, allow user to set new password if token is valid.
